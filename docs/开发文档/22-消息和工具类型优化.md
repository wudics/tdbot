## 1. `todowrite` 工具

### SDK 定义

文件：`packages/opencode/src/tool/todo.ts`（`TodoWriteTool`）

**参数：**

```typescript
const Parameters = Schema.Struct({
  todos: Schema.Array(TodoItem),
})
// TodoItem: { content: string, status: string, priority: string }
// status: "pending" | "in_progress" | "completed" | "cancelled"
// priority: "high" | "medium" | "low"
```

**执行流程：**
1. 请求 `todowrite` 权限
2. 调用 `todo.update({ sessionID, todos })`：
   - 删除该 session 的所有旧待办
   - 插入新待办（带 `position` 排序）
   - 发出 `todo.updated` SSE 事件
3. 返回：
   - `title`: `"${n} todos"`（未完成数量）
   - `output`: JSON prettified todo 数组
   - `metadata`: `{ todos: Todo.Info[] }`

### `todo.updated` SSE 事件

```typescript
{
  type: "todo.updated",
  properties: {
    sessionID: string,
    todos: { content: string, status: string, priority: string }[],
  }
}
```

### 上游展示

**Upstream web app：** `SessionTodoDock` — 可折叠面板，显示 `done/total` 计数、勾选框、已完成项文字划掉、进行中脉冲点。

**TUI：** `[✓]` 已完成、`[•]` 进行中、`[ ]` 待办/取消。

### tdbot 现状（已实现）

- `allTools` 已注册 `todowrite`（`configStore.ts:58`）
- `toolLabel`/`toolLabelRich` 有专门分支：`📋 更新待办 — {n} 个待办`
- `TodoItems` 组件解析 `t.output` JSON，直接渲染待办列表（无折叠）
- 无 `todo.updated` 事件监听（暂不需要，通过 tool output 即可）

### 实现细节

| 模块 | 改动 |
|------|------|
| `src/renderer/src/components/MessageItem.tsx` | `toolLabel`/`toolLabelRich` 添加 `todowrite` 分支；新增 `parseTodoItems` + `TodoItems` 组件；两条渲染路径（parts + legacy）中展开按钮排除 `todowrite`，完成后直接渲染待办列表 |

**渲染格式：**

```
✅ 📋 更新待办 — 3 个待办
  ☑ 编写 API 文档 - high
  ☐ 添加单元测试 - low
  ● 实现用户认证模块 - medium
```

| 项 | 规则 |
|----|------|
| 状态图标 | `☑` completed/cancelled、`●` in_progress、`☐` pending |
| 文本 | `{content} - {priority}`，`line-through` 完成项 |
| 优先级 | 原始值 `high`/`medium`/`low`，不缩写 |
| 展开按钮 | 不显示（待办始终可见） |
| 折叠 | 无（用户要求不需折叠） |

## 2. `task` / `subtask` 部分类型

### TaskTool

文件：`packages/opencode/src/tool/task.ts`

**用途：** 派生子任务给子 agent 执行，支持前台（阻塞）和后台两种模式。

**参数：**

```typescript
{
  description: string,        // 简短描述（3-5 词）
  prompt: string,             // 要执行的任务
  subagent_type: string,      // 子 agent 类型
  task_id?: string,           // 续接已有任务
  command?: string,           // 触发命令
  background?: boolean,       // 后台模式（实验性）
}
```

**输出：** XML 包裹的任务结果 `<task><summary>...<task_result>...</task_result></task>`

### SubtaskPart

文件：`packages/core/src/v1/session.ts`

```typescript
{
  id: PartID,
  sessionID: SessionID,
  messageID: MessageID,
  type: "subtask",
  prompt: string,
  description: string,
  agent: string,
  model?: { providerID: string, modelID: string },
  command?: string,
}
```

### 上游处理

- upstream web app 未跳过 `subtask` type，会渲染显示
- TUI 的 `session.next.step.started` 事件会创建新的 assistant entry，携带 `agent` + `model` 信息

### tdbot 现状（已实现）

- `task` 已列入 `allTools`（已有），`toolLabel` 中 `task`/`subtask` 分支：`📋 {description}`
- `message.part.updated` 事件中 `part.type === 'subtask'` 通过 `stream-subtask` IPC 转发到渲染层
- `SubtaskPart` 类型加入 `ChatPart` 联合，存入 `message.parts[]`
- 思考区域中显示 subtask 条目：`📋 {description} → {agent_name}`

### 实现细节

**渲染效果：**


- `📋` 图标 + 子任务描述（truncate）+ `→ agent 名称`
- Agent 名称优先显示 `name`，未找到回退 `id`（与 agent 标签逻辑一致）

## 3. `file` 部分类型（FilePart）

### SDK 定义

**V1 持久化格式**（`packages/core/src/v1/session.ts`）：

```typescript
{
  id: PartID,
  sessionID: SessionID,
  messageID: MessageID,
  type: "file",
  mime: string,          // "image/png", "application/pdf", "text/plain"
  filename?: string,
  url: string,            // data: URL 或 远程 URL
  source?: FilePartSource, // { type: "file"|"symbol"|"resource", ... }
}
```

**传输格式**（`packages/opencode/src/session/message.ts`）：

```typescript
{
  type: "file",
  mediaType: string,     // 注意：传输格式用 "mediaType" 而非 "mime"
  filename?: string,
  url: string,
}
```

### 产生场景

1. **用户上传附件** — 用户消息中的图片/文档以 `file` part 发送
2. **工具结果附件** — 工具执行返回的图片等媒体以 `file` part 嵌入消息
3. **AI 生成的媒体** — AI 生成图片等也走 `file` part

### 上游展示

**TUI：** 根据 MIME 类型显示不同徽章：
```
[img] screenshot.png  [pdf] report.pdf  [txt] notes.txt  [dir] src/
```

**upstream web app：** 图片内联预览，文档显示文件图标。

### tdbot 现状（已实现）

- `message.part.updated` 事件中 `part.type === 'file'` 会通过 `stream-file` IPC 转发到渲染层
- `useChat.ts` 的 `onFile` 回调将文件数据追加到当前消息的 `message.files[]`
- `message.files` 通过 `saveToStorage` 持久化（已有的 `files` 列）
- assistant 消息正文下方渲染文件列表 + 图片缩略图

### 实现细节

**数据流：**

```
SSE message.part.updated (type: 'file')
  → main/index.ts: IPC 'stream-file' { id, mediaType, filename, url }
    → preload: onFile 桥接
      → useChat.ts: onFile 回调 → messages 内追加到当前 msg.files[]
        → saveToStorage 持久化 → DB files 列
          → MessageItem.tsx: 正文下方渲染
```

**IPC 通道：**

| 方向 | 通道 | 数据 |
|------|------|------|
| main → renderer | `stream-file` | `{ id, mediaType, filename, url }` |

**渲染规则：**

| 条件 | 效果 |
|------|------|
| `image/*` MIME | 行内缩略图 `<img>`，`max-h-[200px] object-contain`，使用原始 URL |
| 其他 MIME | `fileIcon()` 徽章 + 文件名 |
| 位置 | assistant 消息：markdown 正文之后、时间戳之前 |
| 用户文件 | 不变（仍在正文之前） |

**文件变更：**

| 文件 | 改动 |
|------|------|
| `src/main/index.ts` | SSE 循环新增 `part.type === 'file'` 分支，发 `stream-file` IPC |
| `src/preload/index.ts` | 新增 `cleanupFile` + `onFile` 监听器 |
| `src/preload/index.d.ts` | `Api` 接口新增 `onFile` 类型签名 |
| `src/renderer/src/hooks/useChat.ts` | `sendMessage` 内注册 `onFile` 回调，追加到 `message.files` |
| `src/renderer/src/components/MessageItem.tsx` | 用户文件渲染加 `isUser` 条件；assistant 消息正文后新增文件列表渲染（含图片预览） |

## 4. `step-start` / `step-finish` 部分类型

### SDK 定义

**StepStartPart**（`packages/core/src/v1/session.ts`）：

```typescript
{
  id: PartID,
  sessionID: SessionID,
  messageID: MessageID,
  type: "step-start",
  snapshot?: string,
}
```

**StepFinishPart**：

```typescript
{
  id: PartID,
  sessionID: SessionID,
  messageID: MessageID,
  type: "step-finish",
  reason: string,
  snapshot?: string,
  cost: number,
  tokens: {
    total?: number,
    input: number,
    output: number,
    reasoning: number,
    cache: { read: number, write: number },
  },
}
```

### 相关实时事件

**`session.next.step.started`**（实时流事件，比 part 更丰富）：

```typescript
{
  timestamp: number,
  sessionID: string,
  agent: string,
  model: { providerID: string, modelID: string, variant?: string },
  snapshot?: string,
}
```

**`session.next.step.ended`**：

```typescript
{
  timestamp: number,
  sessionID: string,
  finish: string,
  cost: number,
  tokens: { input, output, reasoning, cache: { read, write } },
  snapshot?: string,
}
```

### 上游处理

**upstream web app：** 将 `step-start` 和 `step-finish` 都加入 `SKIP_PARTS` 集合，**不渲染**。它们是消息结构分隔符，不是内容。

**TUI：** 不渲染 part，而是通过 `session.next.step.*` 实时事件驱动 assistant 消息生命周期。

**消息过滤**（`message-v2.ts`）：

```typescript
.filter((msg) => msg.parts.some((part) => part.type !== "step-start"))
```

step-start 用于分隔 assistant 的连续步骤，被过滤掉。

### tdbot 现状（已实现）

- `step-start`：不处理（与 upstream 一致，是结构分隔符）
- `step-finish`：通过 `stream-step-finish` IPC 转发，提取 `cost` 和 `tokens` 存储在 `Message.stepFinish` 字段
- 消息底部靠右渲染：`⚡ {cost.toFixed(6)} | {input}→{output}`
- `step-start` 不渲染，`step-finish` 提取 token/cost 显示在消息底部小字
- 数据通过 `step_finish TEXT` 列持久化到 SQLite

### 实现细节

**IPC 通道：**

| 方向 | 通道 | 数据 |
|------|------|------|
| main → renderer | `stream-step-finish` | `{ id, reason, cost, tokens }` |

**渲染效果：**
```
AI 回复正文……
                              ⚡ 0.000150 | 500→1200
```

- `step-start` 跳过（上游 web app 也在 `SKIP_PARTS` 中跳过）
- `step-finish` 只取最后一条（SDK 内已做累积：`ctx.assistantMessage.cost += usage.cost`）
- `cost` 用 `toFixed(6)` 保留 6 位小数
- `tokens` 显示 `input→output` 格式
- 数据持久化到 SQLite `messages.step_finish` 列

**文件变更：**

| 文件 | 改动 |
|------|------|
| `src/main/index.ts` | SSE 循环新增 `part.type === 'step-finish'` 分支，发 `stream-step-finish` IPC |
| `src/preload/index.ts` | 新增 `cleanupStepFinish` + `onStepFinish` 监听器 |
| `src/preload/index.d.ts` | `Api` 接口新增 `onStepFinish` 类型签名 |
| `src/renderer/src/hooks/useChat.ts` | 新增 `StepFinishInfo` 接口；`Message` 新增 `stepFinish` 字段；`sendMessage` 内注册 `onStepFinish` 回调；`saveToStorage` 包含 `stepFinish` |
| `src/renderer/src/components/MessageItem.tsx` | 消息时间戳下方渲染 `⚡ {cost} \| {input}→{output}` |
| `src/main/sessionStore.ts` | `StoredMessage` 新增 `stepFinish`；`loadMessages` 读取 + `saveMessages` 写入 `step_finish` |
| `src/main/db.ts` | `CREATE TABLE` 新增 `step_finish TEXT` 列 + `ALTER TABLE` 兼容 |

## 5. 滚动卡顿修复

### 问题

滚动消息列表时，跨越 300px 阈值（底部附近）总会卡顿一下。原因：

1. **滚动监听触发 React 重渲染**（`ChatView.tsx:47-59`）：`scroll` 事件中调用 `setShowScrollBtn(!near)`，跨越阈值时触发整个 `ChatView` 组件树的重新协调
2. **useAutoScroll 频繁强制 reflow**（`useAutoScroll.ts:6-14`）：流式传输时 `messages` 频繁变更，每次运行都同步读取 `scrollHeight`/`scrollTop`/`clientHeight`，强制同步布局重算

### 修复

| 方向 | 方案 | 文件 |
|------|------|------|
| **A — 滚动按钮改用 ref** | 移除 `showScrollBtn` state，按钮始终渲染，`scroll` 事件直接操作 `scrollBtnRef.current.style.display` | `ChatView.tsx` |
| **B — rAF 汇聚** | `useAutoScroll` 内用 `requestAnimationFrame` 将 reflow 操作推迟到下一帧；`rafRef` 防同一帧内重复排队 | `useAutoScroll.ts` |

### 改动

**`src/renderer/src/components/ChatView.tsx`：**
- 移除 `showScrollBtn` state + `nearBtnRef`
- 新增 `scrollBtnRef`，按钮始终渲染（`style={{ display: 'none' }}` 初始隐藏）
- `check()` 直接设置 `scrollBtnRef.current.style.display = near ? 'none' : ''`

**`src/renderer/src/hooks/useAutoScroll.ts`：**
- 每次 deps 变更时，若当前帧无排队 rAF，则通过 `requestAnimationFrame` 调度 scroll 判断
- cleanup 取消未执行的 rAF
